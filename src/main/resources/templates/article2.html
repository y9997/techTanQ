はじめに

本記事は、プログラミングを学び始めた方や経験の浅いエンジニア向けに、GitHubの基本から使い方までを解説します。

GitHubは世界中の開発者が利用しているサービスで、ソースコードの管理やチームでの共同開発に欠かせないツールです。
この記事を読むことで、「GitHubとは何か？」から始まり、最低限の操作（環境構築、コミット、プッシュ）ができるようになります。さらに応用編では、チーム開発でよく使われるブランチやPull Requestについても触れていきます。

初めての方でも安心して読み進められるよう、図を多用してわかりやすく説明しますので、ぜひ一緒に学んでいきましょう！

基本概要

Githubとは

Githubは、ソフトウェア開発のためのWebベースのプラットフォームで、ソースコードの管理や共有、バージョン管理を簡単に行うことができます。

Gitという分散型バージョン管理システムをベースにしており、チーム開発はもちろん個人開発にも広く利用されています。
GitHubを使えば、コードの履歴を追ったり、他の開発者と効率的に連携したり、修正内容をレビューして反映するなど、開発の質とスピードを大きく向上させることができます。

現在、GitHubのユーザー数は1億人を超えており、世界中の開発現場で活用されています。
プログラミングを生んでいる方やエンジニアを目指す方にとって、GitHubのスキルは必須とも言えるでしょう。

リポジトリとは

リポジトリとは、コード、ファイル、各ファイルの改訂履歴を保存、管理する場所のことです。GitHubでは、このリポジトリを使ってプロジェクトの状態を記録し、過去の変更を追跡することができます。

リポジトリにはリモートリポジトリとローカルリポジトリの2種類があります。
リモートリポジトリとはGitHub上に存在し、複数人の開発者と共同で使用するリポジトリのことです。
チーム開発やオープンソースプロジェクトなどで活用されます。

ローカルリポジトリとはローカル環境に存在し、個人で作業する際に使用されます。
ここで行った変更を後でリモートリポジトリに反映させることができます。

GitHubではこの２つのリポジトリを連携することで、チーム開発やバージョン管理を可能にしています。

用語解説

GitHubを使う上で頻繁に登場する基本用語を初心者向けにわかりやすく解説します。

ステージ
リモートリポジトリに反映したいファイルを選び、変更を記録する準備をすることを「ステージ」と呼びます。
この操作によって、どのファイルを次のコミットに含めるかを指定できます。

コミット
ステージしたファイルの変更内容を、ローカルリポジトリに保存する操作です。
「いつ」「何を変更したか」を記録することで、履歴を追えるようになります。

プッシュ
ローカルリポジトリに保存された変更をリモートリポジトリに反映させる操作です。
これにより、他の開発者と変更内容を共有できます。

ブランチ
リポジトリに中で、機能追加や修正などを行うために分岐させた作業用の領域です。
本流(main)とは別に作業できるため、安全に開発を進められます。

クローン
リモートリポジトリの内容を、自分のローカル環境にコピーする操作です。
これにより、手元で編集や確認ができるようになります。

フォーク
他人のリモートリポジトリをコピーして、自分のGitHubアカウント常に新しいリモートリポジトリを作成する操作です。
オープンソースプロジェクトへの貢献や、独自の改良を加える際に使われます。

基本操作

この章では、GitHubを使い始めるための環境構築から基本的な操作方法までわかりやすくまとめました。
個人開発に必要なスキルが身につく構成になっていますので、ぜひ本章を参考に実際に手を動かしてみてください。

環境構築

Gitのインストール
GitHubはGitに依存しているため、ローカル環境にGitが存在している必要があります。
ここでは、OS別にGitのインストール方法を紹介します。

macOS
1.まずはターミナルを開き、Gitがインストールされているか以下のコマンドで確認してください。 　　　

git --version

バージョンが表示されない場合は、以下の手順でインストールしてください。

2.homebrewでインストールします。以下のコマンドを実行してください。 　　　

brew install git

3.再度バージョン確認を行い、表示されればインストール完了です。 

git --version


windows
1.以下の公式ページからwindows版Gitをダウンロードしてインストールします。
Git Downloads

2.インストール後、コマンドプロンプトを開いて以下のコマンドを実行します。 　　　

git --version
　　　

バージョン情報が表示されたら成功です。




GitHubアカウント作成
GitHubを利用するには、アカウントが必要です。
以下のGitHub公式サイトにアクセスし、アカウントを新規作成してください。
Git公式サイト

リポジトリ作成と連携

ここでは、GitHub上のリモートリポジトリと、PC上のローカルリポジトリの作成と連携方法について解説します

リモートリポジトリの作成
GitHubの公式サイトから、GUI操作で簡単に作成できます。

1.以下のURLからGitHubにアクセスし、ログインしてください。
Git公式サイト


2.画面右上の＋ボタンからNew repositoryを選択します。
　キャプチャ

3.各項目を入力し、create repositoryボタンをクリックします。
　キャプチャ

これでリモートリポジトリに作成は環境です。


ローカルリポジトリの作成
ローカルリポジトリは、任意のディレクトリに.gitフォルダを作成することで構築できます。
このgitフォルダにはリモートリポジトリとの接続情報や、ファイルの変更履歴などが保存されます。

ローカルリポジトリを作成したいディレクトリから以下のコマンドを実行します。 　　　

git init

以下のコマンドを実行して、.gitフォルダが混在するか確認を行います。

ls -a

.gitが表示されたらローカルリポジトリの作成は完了です。
表示されない場合は再度git initを実行してください。




リポジトリ同士を繋げる
現段階では各リポジトリは独立している状態なので、繋げる必要があります。
以下のコマンドを実行します。 

git remote add origin https://github.com/ユーザー名/リポジトリ名.git

連携が成功したか確認するには以下のコマンドを実行してください。

git remote -v

下記のように表示されたら成功です。 

origin https://github.com/ユーザー名/リポジトリ名.git(fetch)
origin https://github.com/ユーザー名/リポジトリ名.git(push)

プッシュ

前項でリポジトリの作成と連携ができたので、ファイルをプッシュしてみましょう。
プッシュするには、ステージ→コミット→プッシュの手順を踏む必要があります。
一つ一つ確実に実行していきましょう。

1.ステージする
　変更や追加したファイルを「次の作業対象」として登録する操作を行います。
　.gitが存在するディレクトリから以下のコマンドを実行します。 　　　

git add .

　

特定のファイルだけをステージしたい場合は、以下のように指定します。

git add ファイル名

2.コミットする
　ステージしたファイルを「変更履歴」として記録します。
以下のコマンドを実行します。メッセージは変更内容がわかるように書くと良いです。 

git commit -m "first commit"

3.プッシュする
　以下のコマンドを実行します。
　ローカルの変更をGitHub上のリポジトリに反映させます。

　　　
git push

4.確認する
プッシュが成功したか確認するにはGitHubにアクセスします。
Git公式サイト

自身のアカウント→対象のリポジトリを開いてください。
ファイルが表示されたら成功です。

応用編

この章ではチーム開発が可能なスキルを身につけることができるようまとめました。
本章を理解することで初心者から脱却しましょう。

クローンとプル

クローンとプルはチーム開発において必須の知識とスキルです。

まずはクローンをおさらいしましょう。
クローンはリモートリポジトリの内容を自分のローカル環境にコピーする操作です。
リモートリポジトリの内容を丸ごとコピーするので最初の１回だけ行います。

プルはリモートリポジトリの最新の変更をローカルに取り入れることです。
変更だけなので２回目以降、他の人が更新した時などに行います。

それでは実際に操作してみましょう。

クローン
以下のコマンドを実行しクローンします。 

git clone https://github.com/ユーザー名/リポジトリ名.git
　　

クローンが成功したか確認するため以下のコマンドを実行します。

　　
ls -a

リポジトリ名と同じ名前のフォルダが作成されていれば成功です。
そのフォルダ内にクローンしたファイル群が入っています。

プル
以下のコマンドを実行しプルします。

git pull origin main

originはリモートリポジトリの名前(通常はorigin)、mainはブランチ名です。

実行後、以下のようなメッセージが表示されれば成功です。 

remote:Enumerating objects:...
remote:Counting objects:...

Updating abc1234..def5678
Fast-forward
ファイル名｜変更行数

ブランチとマージ

ブランチとマージはチーム開発を安全に行うために重要な技術です。

まず、ブランチのおさらいをします。
ブランチとは、リポジトリの中で機能追加や修正などを行うために分岐させた作業用の領域です。

マージとは、ブランチで分岐させた作業内容をmainブランチなどの本流に統合する操作です。

ブランチで作業用の領域を作り機能追加などを行った後にマージを行うことで、安全に開発を行うことができます。

それでは実際の操作を説明します。

ブランチ
以下のコマンドを実行します。 

git branch ブランチ名

ブランチができたのか確認するには以下のコマンドを実行します。

git branch

作成したブランチ名が表示されていればブランチの作成は成功です。



マージ
まず、マージ先のブランチに切り替えます。(例：mainブランチ)

　
git switch main

その後、以下のコマンドを実行して作業ブランチを統合します。

git merge ブランチ名

マージが成功すると以下のようなメッセージが表示されます。

　　　
Updating abc1234..def5678
Fast-forward
ファイル名｜変更行数

または

merge mode by the 'recursibe' strategy.
ファイル名｜変更行数

もし以下のメッセージが表示される場合はコンフリクトが起きています。

Auto-merging ファイル名
CONFLICT(content):Merge conflict in ファイル名

コンフリクトとは、両方のブランチで同じファイルが返納されていた場合、GItがどちらの変更を適用するのか判断できず、マージできないことです。
例えば、mainブランチをfeatureブランチの両方でexample.htmlの同じ行を編集した場合、Gitはどちらを採用すべきか判断できず、コンフリクトが発生します。
この場合は、手動でファイルを修正し、以下のコマンドで完了させます。

git add ファイル名
git commit

コンフリクトが起きた時点で、Gitはマージプロセスを一時停止の状態になっています。
修正後にコミットするだけでGitはマージが完了したと認識するため、再度git merggeを実行する必要はありません。

IssueとPull Request

IssueとPull Requestを使いこなせれば、効率よくチーム開発を行うことができます。

Issueとは、例えば「ログイン画面でエラーが出る」「検索機能を追加したい」といった課題や提案を記録・共有するための機能です。
チーム開発のコミュニケーションやタスク管理に役立ちます。

一方でPull Requestとは、Issueの対応を実装した成果物です。
業務でチーム開発を行う際はIssueで課題やタスクなどを共有し、その成果物をPull Requestで報告する流れが一般的です。
Pull Requestされた成果物はレビューされ、問題なければマージされます。
このように、IssueとPull Requestはチーム開発においてより円滑にするために使用されます。
